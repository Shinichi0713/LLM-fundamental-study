## RLHFとは

RLHF（Reinforcement Learning from Human Feedback：人間からのフィードバックによる強化学習）は、AIモデルを **「人間の価値観や好みに沿うように」** チューニングするための極めて重要なプロセスです。

単に知識を詰め込むだけでなく、AIに「どう振る舞うのが望ましいか」を教え込むための、いわば **「AIのしつけ」** といえる仕組みです。

### 1. RLHFの3つの基本ステップ

RLHFは一般的に以下の3つのプロセスを経て行われます。

1. **SFT（教師あり微調整）:**
まず、人間が書いた高品質な「質問と模範回答」のペアを使って、AIを学習させます。これでAIは「指示に従う」というベースの能力を獲得します。
2. **報酬モデルの学習（Reward Modeling）:**
AIが生成した複数の回答を人間が比較・評価（ランク付け）します。「Aの方がBより役に立つ」「Cは不適切」といった評価データを学習させ、**「人間が何を好むか」を判定する別のAI（報酬モデル）**を構築します。
3. **強化学習（RL）による微調整:**
報酬モデルを「試験官」にしてAIを特訓します。AIが生成した回答に対して、報酬モデルがスコアを付け、そのスコアが高くなるように（＝人間に好まれる回答になるように）AIのパラメータを更新していきます。

### 2. なぜRLHFが必要なのか？

大規模言語モデルは、インターネット上のあらゆるテキストを学習していますが、その中には誤情報、偏見、有害なコンテンツも含まれています。

* **意図との一致（Alignment）:** AIが「有害なことを言わない」「指示通りに誠実に答える」ためには、単なる予測学習だけでは不十分です。RLHFは、AIが人間の期待する「礼儀」や「安全性」を内面化させるために不可欠です。
* **好みの反映:** 複数の正解がある場合、人間にとって「分かりやすい」「親切である」というニュアンスをAIに学習させることができます。

### 3. RLHFが直面している課題と限界

非常に強力な手法ですが、いくつか克服すべき課題もあります。

* **コストの増大:** 人間が一つひとつの回答を評価する必要があるため、膨大な時間とコストがかかります。
* **報酬のハッキング:** AIが「人間が好むような、もっともらしい嘘」をついて報酬スコアを稼ぐ、いわゆる「AIハック」が起きることがあります。
* **主観性の問題:** 評価する人によって「何が正しい回答か」が異なるため、特定の文化や価値観に偏ってしまう可能性があります。

### 4. 最新のトレンド：RLHFの進化形

現在、RLHFをより効率的に行う研究が急速に進んでいます。

* **RLAIF (Reinforcement Learning from AI Feedback):**
人間が評価する代わりに、極めて賢いAI（GPT-4やClaudeなど）を評価者として使い、AIがAIを育てる手法です。
* **Direct Preference Optimization (DPO):**
複雑な強化学習のステップ（PPOなど）を省略し、直接「選ばれた回答」と「選ばれなかった回答」の差分から学習を行うことで、安定して高性能なモデルを作る手法です。


### まとめ

| プロセス | 役割 |
| --- | --- |
| **SFT** | 基礎体力作り（回答の型を学ぶ） |
| **報酬モデル** | 好みの言語化（何が良い回答かを教える） |
| **強化学習** | 改善の反復（報酬を最大化する振る舞いを身につける） |

## アーキテクト

Google Colabの限られた環境（無料枠で約12〜15GBのVRAM）でRLHFを実装するためのアーキテクチャは、 **「計算リソースの極小化」と「学習効率の最大化」** が全てです。

本格的なPPOのフル実装はメモリ不足で即座にクラッシュするため、以下のコンポーネントで構成されるアーキテクチャが現実的です。


### Google Colab用 RLHF 実装アーキテクチャ

この構成は、メモリ消費を抑えつつ、LLMに「学習」させるための最小構成単位です。

#### 1. 学習対象のモデル (Policy Model)

* **構成:** `Base Model + LoRA Adapters`
* **仕組み:** ベースモデル（7Bパラメータ以下の量子化モデル）はメモリに固定（Frozen）し、**LoRA** を用いて少数のパラメータのみを更新します。これにより、必要なVRAMを数GB単位で節約できます。

#### 2. 報酬モデル (Reward Model)

* **構成:** 評価用LLM (Reward Model)
* **役割:** 学習中のモデル（Policy）が生成した回答に対して、「人間の好み（またはルール）」に基づいたスコア $r$ を返します。
* **Colabでの工夫:** 巨大な報酬モデルを別途ロードするとメモリが足りないため、**「学習対象モデルと報酬モデルを共有」**するか、あるいは**「ルールベースの関数（キーワードの有無や計算結果の正誤）」**で報酬を計算する手法が推奨されます。

#### 3. リファレンスモデル (Reference Model)

* **役割:** PPOアルゴリズムにおいて、「学習しすぎてモデルが元の言語能力を失う（崩壊する）」のを防ぐためのブレーキ役。
* **実装:** ベースモデルをそのままコピーしてメモリに置きます。これもLoRAのみを動かすことでVRAMを抑制します。

### 推奨される実装ステップと構成図

| コンポーネント | Colabでの実装手法 | メモリ節約術 |
| --- | --- | --- |
| **Policy Model** | `trl.AutoModelForCausalLMWithValueHead` | **LoRA + 4bit/8bit量子化** |
| **Reward Model** | `RewardModel` または `RewardFunction` | 巨大なモデルは使わず**ルールベース**で代用 |
| **Ref Model** | `Base Model` | 更新しないため**軽量化（または共有）** |


### 実装上の注意点と「軽量化」のコツ

* **メモリ効率化の鉄則:**
`trl` ライブラリを使用する際、`is_peft_model=True` を設定し、`ValueHead`（PPOで報酬を予測する層）だけが学習されるように調整してください。
* **データセットの工夫:**
RLHFのステップで使うデータセットは、あらかじめ「質問(prompt)」のみのリストを用意し、学習中にモデルが回答を生成するようにします。Colabでは **batch_size=1** が基本です。
* **PPOよりも軽量な手法:**
もしPPOが重すぎると感じるなら、**DPO (Direct Preference Optimization)** を検討してください。DPOは強化学習の複雑なループ（報酬モデルやリファレンスモデルの更新）を必要とせず、好みのペアデータがあれば教師あり学習の延長で強化学習相当の効果が得られるため、ColabでのRLHFには最も適しています。
