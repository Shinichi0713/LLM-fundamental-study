目次

    DPO（Direct Preference Optimization）は、報酬モデルとPPOを使わずに、選好データから直接モデルを最適化する手法です。RLHFの複雑さを避けつつ、人間の好みに沿った出力を得ることを目指します。

    概要

    1. 直感的なアイデア

    RLHFでは通常：

    1. 人間の選好データから報酬モデルを学習
    2. その報酬モデルを最大化するように、PPOでモデルを更新

    という2段階が必要で、安定性や実装の難しさが課題でした。

    DPOはこれを1段階にまとめます：

    • 「モデル自身が、実は暗黙的な報酬モデルになっている」とみなす
    • 選好データ（「こっちの出力の方が好ましい」というペア）から、直接モデルのパラメータを更新する

    2. データ形式

    DPOで使うデータは、典型的には次のようなタプルです：

    • 入力（プロンプト）：xxx
    • 好ましい出力（chosen）：ywy_wyw​
    • 好ましくない出力（rejected）：yly_lyl​

    データセット：

    D={(x(i),yw(i),yl(i))}i=1N\mathcal{D} = \{ (x^{(i)}, y_w^{(i)}, y_l^{(i)}) \}_{i=1}^N D={(x(i),yw(i)​,yl(i)​)}i=1N​

    3. 目的関数（数式レベル）

    3.1 RLHF（PPO）の目的

    RLHFでは、だいたい次のような形で報酬最大化＋KL正則化を行います：

    max⁡πθEx∼D,y∼πθ(y∣x)[rϕ(x,y)]−β KL(πθ(y∣x) ∥ πref(y∣x))\max_{\pi_\theta} \mathbb{E}_{x \sim \mathcal{D}, y \sim \pi_\theta(y|x)} \big[ r_\phi(x, y) \big] - \beta \, \text{KL}\big( \pi_\theta(y|x) \,\|\, \pi_{\text{ref}}(y|x) \big) πθ​max​Ex∼D,y∼πθ​(y∣x)​[rϕ​(x,y)]−βKL(πθ​(y∣x)∥πref​(y∣x))

    -πθ\pi_\thetaπθ​：学習中のポリシー（モデル） -πref\pi_{\text{ref}}πref​：参照モデル（元のSFTモデルなど） -rϕr_\phirϕ​：別途学習した報酬モデル -β\betaβ：正則化の強さ

    3.2 DPOの目的関数

    DPOでは、報酬モデルrϕr_\phirϕ​を明示的には持たず、代わりに「モデルの対数確率の差」を「報酬の差」とみなします。

    DPOの損失関数は、おおまかに次のように書けます：

    LDPO(πθ;πref)=−E(x,yw,yl)∼D[log⁡σ(β(log⁡πθ(yw∣x)πref(yw∣x)−log⁡πθ(yl∣x)πref(yl∣x)))]\mathcal{L}_{\text{DPO}}(\pi_\theta; \pi_{\text{ref}}) = -\mathbb{E}_{(x, y_w, y_l) \sim \mathcal{D}} \left[ \log \sigma \left( \beta \left( \log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)} \right) \right) \right] LDPO​(πθ​;πref​)=−E(x,yw​,yl​)∼D​[logσ(β(logπref​(yw​∣x)πθ​(yw​∣x)​−logπref​(yl​∣x)πθ​(yl​∣x)​))]

    ここで：

    -πθ(y∣x)\pi_\theta(y|x)πθ​(y∣x)：学習中のモデルが出力yyyを生成する確率 -πref(y∣x)\pi_{\text{ref}}(y|x)πref​(y∣x)：参照モデルが出力yyyを生成する確率 -β\betaβ：温度パラメータ（報酬のスケールを調整） -σ\sigmaσ：シグモイド関数

    直感的な意味：

    • 括弧の中身

    log⁡πθ(yw∣x)πref(yw∣x)−log⁡πθ(yl∣x)πref(yl∣x) \log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)} logπref​(yw​∣x)πθ​(yw​∣x)​−logπref​(yl​∣x)πθ​(yl​∣x)​

    は、「好ましい出力ywy_wyw​の相対的な対数確率」から「好ましくない出力yly_lyl​の相対的な対数確率」を引いたもの。

    • これが報酬の差rϕ(yw,x)−rϕ(yl,x)r_\phi(y_w,x) - r_\phi(y_l,x)rϕ​(yw​,x)−rϕ​(yl​,x)に対応すると解釈できます。
    • シグモイドを通すことで、「好ましい出力の方がより高い報酬を持つ」ように学習します。

    つまり、報酬モデルを学習せずに、モデルの確率比から直接「どちらが好ましいか」を学習していることになります。arXiv

    4. 学習の流れ（実装イメージ）

    1. 参照モデルπref\pi_{\text{ref}}πref​を用意

      • 通常はSFT（Supervised Fine-Tuning）済みモデルを使います。

    2. 選好データセットD\mathcal{D}Dを用意

      • 各サンプルは(x,yw,yl)(x, y_w, y_l)(x,yw​,yl​)の形式。

    3. DPO損失で勾配降下

      • 各バッチについて、πθ(yw∣x),πθ(yl∣x),πref(yw∣x),πref(yl∣x)\pi_\theta(y_w|x), \pi_\theta(y_l|x), \pi_{\text{ref}}(y_w|x), \pi_{\text{ref}}(y_l|x)πθ​(yw​∣x),πθ​(yl​∣x),πref​(yw​∣x),πref​(yl​∣x)を計算
      • 上記のDPO損失を計算し、πθ\pi_\thetaπθ​のパラメータを更新 -πref\pi_{\text{ref}}πref​は固定（勾配を流さない）

    4. 終了条件

      • 一定エポック数、または検証セットの性能が頭打ちになったら終了。

    実務では、Hugging Faceの trl ライブラリなどでDPOが実装されており、比較的簡単に試せます。Hugging Face Blog

    5. RLHF（PPO）との違い

    項目 RLHF（PPO） DPO
    報酬モデル 明示的に学習・保持 暗黙的（モデル自身が報酬モデル）
    強化学習 PPOなどのRLアルゴリズムが必要 勾配降下のみ（教師あり学習に近い）
    安定性 学習が不安定になりやすい 比較的安定
    実装の複雑さ 高い（報酬モデル＋PPO） 低い（損失関数を定義して最適化）
    メモリ使用量 報酬モデル分が増える 参照モデルは固定で軽量

    6. 利点と注意点

    利点

    • パイプラインがシンプル：報酬モデル学習＋PPOが不要
    • 安定性が高い：RLHFで起きがちな学習の不安定さを軽減
    • 実装が容易：既存のSFTコードに近い形で実装できる
    • メモリ効率が良い：報酬モデルを保持しない

    注意点（欠点）

    • 分布シフトに弱い：学習データと実際の利用分布が大きく異なると性能が落ちやすい
    • 探索が弱い：PPOのような明示的な探索がないため、局所解に陥りやすい
    • データ品質に敏感：ノイズの多い選好データだと性能が悪化しやすい
    • 長さバイアス：長い回答が好まれやすいなどのバイアスが入りやすい（これを補正するSamPOなどの改良版も提案されています）Hugging Face Blog

    仕組み

    DPO（Direct Preference Optimization）の仕組みを、できるだけ直感的に説明します。

    1. ざっくり言うと

    DPOは、

    「どちらの出力が好ましいか」という人間の選好データだけを使って、
    報酬モデルや強化学習（PPO）を使わずに、モデルを直接更新する手法

    です。

    RLHFでは「報酬モデルを学習 → PPOで強化学習」という2段階が必要でしたが、DPOはそれを1段階で済ませるのがポイントです。

    2. データの形

    DPOで使うデータは、だいたい次のようなペアです：

    • プロンプト（質問）：xxx
    • 好ましい回答（chosen）：ywy_wyw​
    • 好ましくない回答（rejected）：yly_lyl​

    例： -xxx：「日本の首都は？」 -ywy_wyw​：「日本の首都は東京です。」 -yly_lyl​：「日本の首都は大阪です。」

    こういう「どっちが良いか」のペアをたくさん集めたデータセットを使います。

    3. モデルが「暗黙の報酬モデル」になる

    DPOの一番大事なアイデアは：

    モデル自身が、実は「報酬モデル」の役割をしている

    とみなすことです。

    具体的には：

    • 参照モデル（元のSFTモデルなど）をπref\pi_{\text{ref}}πref​とします。
    • 学習中のモデルをπθ\pi_\thetaπθ​とします。

    このとき、

    log⁡πθ(y∣x)πref(y∣x)\log \frac{\pi_\theta(y|x)}{\pi_{\text{ref}}(y|x)} logπref​(y∣x)πθ​(y∣x)​

    という「モデルの確率と参照モデルの確率の比（対数）」を、その回答の報酬とみなします。

    • この値が大きい → その回答は「好ましい」と判断されやすい
    • この値が小さい → その回答は「好ましくない」と判断されやすい

    4. 学習のイメージ

    DPOの学習は、次のような「ルール」でモデルを更新します：

    好ましい回答ywy_wyw​の相対確率が、好ましくない回答yly_lyl​の相対確率より大きくなるようにする

    数式で書くと、次の差：

    log⁡πθ(yw∣x)πref(yw∣x)−log⁡πθ(yl∣x)πref(yl∣x)\log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)} logπref​(yw​∣x)πθ​(yw​∣x)​−logπref​(yl​∣x)πθ​(yl​∣x)​

    が大きくなるようにモデルを更新します。

    • この差が大きいほど、「ywy_wyw​の方がyly_lyl​より好ましい」という意味になります。
    • DPOの損失関数は、この差をシグモイド関数で評価し、「好ましい方の確率が高くなる」ように最適化します。

    つまり、報酬モデルを別途学習せずに、モデルの確率比から直接「どちらが好ましいか」を学習していることになります。arXiv

    5. 学習の流れ（ざっくり）

    1. 参照モデルπref\pi_{\text{ref}}πref​を用意

      • 普通はSFT済みのモデルを使います。

    2. 選好データ(x,yw,yl)(x, y_w, y_l)(x,yw​,yl​)を用意

    3. DPOでモデルを更新

      • 各サンプルについて、モデルがywy_wyw​とyly_lyl​を出す確率を計算
      • 参照モデルも同じyw,yly_w, y_lyw​,yl​について確率を計算
      • 上記の「確率比の差」が大きくなるように、モデルのパラメータを更新
      • 参照モデルは固定（更新しない）

    4. 繰り返す

      • これを何エポックか繰り返すと、モデルは「好ましい回答をより出しやすく、好ましくない回答を出しにくく」なります。

    6. RLHFとの違い（イメージ）

    • RLHF（PPO）

      • 人間の選好 → 報酬モデル学習 → PPOで強化学習
      • 報酬モデルとPPOの2つが必要で、学習が不安定になりやすい

    • DPO

      • 人間の選好 → モデルの確率比を「報酬」とみなして直接更新
      • 報酬モデルもPPOも不要で、教師あり学習に近い形で学習できる

    つまり、「報酬モデル＋PPO」を「モデル自身の確率比」に置き換えたのがDPOの仕組みです。arXivHugging Face Blog

    まとめ

    • DPOは、人間の「どちらが良いか」という選好データだけを使って、モデルを直接最適化する手法です。
    • モデルの確率と参照モデルの確率の比を「報酬」とみなし、好ましい回答の相対確率が高くなるように学習します。
    • これにより、報酬モデルやPPOを使わずに、RLHFと同等のアライメント効果を、よりシンプルで安定した形で実現できます。
