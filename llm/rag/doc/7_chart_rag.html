目次

    RAGを行う場合、技術資料に掲載されている画像と関連する文章を両方参照したい(マルチモーダルRAGと呼ばれます)ということがあります。 それも結構な頻度で。

    本日テーマ：

    画像を含めてRAGを実現する方法について実現法を整理します

    概要

    文章と画像を一緒に検索したい場合、代表的なアプローチは大きく次の4つに分けられます。

    1. マルチモーダル埋め込みモデルを使う（CLIP系）

    CLIP（Contrastive Language-Image Pretraining）のようなマルチモーダル埋め込みモデルを使う方法です。

    • テキストと画像を同じベクトル空間に埋め込む
    • テキストクエリで画像を検索したり、画像クエリでテキストを検索したりできる
    • 例：OpenAI CLIP、JinaCLIP、Apple DFN5B-H、Meta系モデルなどPineconeTowards Data Science

    RAGへの組み込みイメージ

    • 文書中のテキストと画像をCLIPで埋め込み、同じベクトルDBに保存
    • ユーザーのテキストクエリ → テキスト埋め込み → ベクトル検索 → テキスト・画像の両方を取得
    • 取得したテキスト＋画像をマルチモーダルLLM（GPT-4Vなど）に渡して回答生成

    メリット

    • テキストと画像を同じ空間で比較できるため、セマンティックな検索が可能
    • 画像そのものをベクトルとして扱える

    デメリット

    • モデルやインフラの準備が必要（CLIPのホスト、ベクトルDBなど）

    2. 画像をテキスト化してRAGに組み込む

    画像をテキストで記述し、それをRAGの文書として扱う方法です。

    • GPT-4Vなどの画像理解モデルで画像を詳細に説明（キャプション生成）
    • そのテキストを通常のテキストRAGパイプラインに流す
    • 検索時は「テキストクエリ vs 画像キャプション」で類似度を計算

    RAGへの組み込みイメージ

    • 画像 → GPT-4Vで説明文生成 → テキストとしてベクトル化・保存
    • 検索時はテキストクエリでキャプションを検索し、対応する画像も一緒に取得
    • LLMに「テキスト＋画像URL」を渡して回答生成

    メリット

    • 既存のテキストRAGパイプラインをほぼそのまま流用できる
    • 画像専用の埋め込みモデルが不要

    デメリット

    • 画像の細かい視覚情報がテキスト化で失われる可能性
    • GPT-4VのAPIコストがかかる

    3. ハイブリッド検索（テキスト＋画像のマルチベクトル）

    マルチベクトル・リトリーバーのように、テキストと画像を別々に埋め込み、検索時に統合する方法です。

    • テキスト用の埋め込み（例：OpenAI text-embedding-3-small）
    • 画像用の埋め込み（例：CLIP）
    • それぞれをベクトルDBに保存し、検索時にスコアを統合（例：Reciprocal Rank Fusion）

    LangChainの「Multi-Vector Retriever」などがこの考え方に近いですLangChain Blog。

    メリット

    • テキストと画像で最適なモデルを選べる
    • スコアの重み付けを調整しやすい

    デメリット

    • 実装がやや複雑（複数の埋め込み・インデックス管理）

    4. クラウドサービスのマルチモーダル検索機能を使う

    Azure AI SearchやOpenSearchなど、マルチモーダル検索をネイティブにサポートするサービスを使う方法です。

    • Azure AI Search：テキスト＋画像を含むドキュメントをインポートし、マルチモーダル埋め込みで検索Microsoft Learn
    • OpenSearch：Titan Multimodal Embeddingsなどを使い、テキスト・画像・（テキスト＋画像）クエリをサポートOpenSearch Blog

    メリット

    • インフラ構築が比較的簡単
    • スケーラビリティや運用面の負担が少ない

    デメリット

    • サービス固有の制約（料金、モデル選択の自由度など）

    課題1: 複数物体が映る画像の扱い

    CLIP系のマルチモーダル埋め込みは「画像全体」を一つのベクトルにまとめるため、画像内に複数の対象が写っていると、特定の対象だけを狙った検索が難しくなるという問題があります。

    なぜCLIPだけでは弱いのか

    • CLIPは画像全体を一つの埋め込みベクトルに圧縮します。
    • そのため、「画像内のどの部分がクエリに対応しているか」を細かく制御できません。
    • 複数の対象が写っている画像では、
      • 「犬と猫が写っている画像」に対して「犬だけ」を検索したい
      • 「複数の商品が写っている画像」に対して「特定の商品だけ」を検索したい といった部分的なマッチングが苦手になります。

    対応策：大きく分けて3つのアプローチ

    1. 画像を分割して埋め込む（Region-level / Patch-level embedding）

    画像を領域ごとに分割し、それぞれを別々に埋め込む方法です。

    • 例：
      • グリッド分割（例：3×3や4×4のパッチ）
      • 物体検出（YOLOなど）でバウンディングボックスごとに切り出し
    • 各パッチ／領域をCLIPで埋め込み、複数のベクトルとして保存します。
    • 検索時は、クエリと各パッチのベクトルを比較し、最もスコアの高いパッチに対応する画像をヒットさせます。

    メリット

    • 「画像全体」ではなく「画像の一部」を対象にした検索が可能
    • 物体検出を使えば、意味のある領域単位で検索できる

    デメリット

    • ベクトル数が増えるため、ストレージ・検索コストが増加
    • どのように分割するか（グリッドサイズ・物体検出の閾値など）のチューニングが必要

    2. マルチモーダルLLMで「対象ごとのキャプション」を生成する

    GPT-4VなどのマルチモーダルLLMを使って、

    • 「この画像に写っている各対象を、それぞれ詳しく説明して」
    • 「商品A、商品B、商品Cごとにキャプションを生成して」

    といった指示で、対象ごとのテキスト説明を生成します。

    その後、

    • 各キャプションをテキスト埋め込み（例：OpenAI text-embedding-3-small）でベクトル化
    • 検索時はクエリと各キャプションのベクトルを比較

    という形で、テキストベースのRAGとして扱います。

    メリット

    • 対象ごとの「意味的な説明」をテキストで持てるため、検索精度が上がりやすい
    • 既存のテキストRAGパイプラインをほぼそのまま使える

    デメリット

    • GPT-4VなどのAPIコストがかかる
    • 画像の細かい視覚情報がテキスト化で失われる可能性

    3. マルチベクトル検索＋リランキング

    複数の埋め込み（画像全体・領域ごと・キャプションなど）を組み合わせる方法です。

    • 例：
      • 画像全体のCLIP埋め込み
      • 物体ごとのCLIP埋め込み
      • 各物体のテキストキャプションの埋め込み
    • これらを別々のベクトルとして保存し、検索時に
      • 各ベクトルに対してスコアを計算
      • Reciprocal Rank Fusion（RRF）などでスコアを統合
      • 必要に応じて**リランキング（クロスエンコーダなど）**で順位を調整

    メリット

    • 「画像全体の文脈」と「特定の対象」の両方を考慮できる
    • リランキングを組み合わせることで、より高精度な順位付けが可能

    デメリット

    • 実装がやや複雑（複数の埋め込み・インデックス管理）

    課題2: 画像と文章の埋め込み表現の不均一

    画像とテキストの埋め込みは、モデルや学習データが異なるため、同じベクトル空間でも「距離感」や「品質」が不均一になりがちです。この問題への対応策を、実務で使えるレベルまで整理します。

    1. なぜ「距離・品質の不均一」が起きるのか

    • 学習タスクの違い
      テキスト埋め込みは「文書検索・意味類似度」を主目的に学習されることが多く、
      画像埋め込み（CLIPなど）は「画像–テキストの対応付け」を目的に学習されます。
    • スケール・分布の違い
      テキスト埋め込みと画像埋め込みで、
      • ベクトルのノルム分布
      • 類似度スコアのレンジ
        が異なるため、同じ距離尺度で比較すると片方に偏ることがあります。
    • モデル間の性能差
      テキスト用の最新モデル（例：OpenAI text-embedding-3-large）と、
      画像用のモデル（例：CLIP ViT-B/32）で性能差が大きいと、片方の埋め込みが「支配的」になり、検索結果が偏ります。

    2. 対応策①：正規化とスケーリングで「距離感」を揃える

    ベクトルの正規化（Normalization）

    • テキスト・画像の埋め込みをL2正規化して、
      • すべてのベクトルを単位球面上に配置
      • 内積＝コサイン類似度として扱う
        ことで、ノルムの違いによる偏りを減らせますOpenAI Cookbook。
    # 例：OpenAIのtext-embedding-3-small / CLIP埋め込みを正規化
import numpy as np

def normalize(emb):
    norm = np.linalg.norm(emb)
    return emb / norm if norm > 0 else emb

text_emb_norm = normalize(text_emb)
image_emb_norm = normalize(image_emb)

    スコアのスケーリング（Score Scaling）

    • テキスト–テキスト、テキスト–画像の類似度スコアの分布を事前に計測し、
      • 平均・分散を揃えるように線形変換（例：z-score scaling）
      • あるいはmin–maxスケーリング
        を行い、スコアのレンジを揃えます。
    # 例：テキスト–テキスト / テキスト–画像スコアの分布を揃える
# μ_tt, σ_tt: テキスト–テキストスコアの平均・標準偏差
# μ_ti, σ_ti: テキスト–画像スコアの平均・標準偏差

def scale_score(score, source_mean, source_std, target_mean, target_std):
    z = (score - source_mean) / source_std
    return z * target_std + target_mean

# テキスト–画像スコアをテキスト–テキストの分布に合わせる
scaled_score = scale_score(
    raw_ti_score, μ_ti, σ_ti, μ_tt, σ_tt
)

    3. 対応策②：マルチベクトル検索＋スコア統合

    Reciprocal Rank Fusion（RRF）

    • テキスト検索結果と画像検索結果を別々にランキングし、
      RRFで順位を統合しますElasticsearch Docs。
    # RRFの簡易実装例
def rrf(ranks_list, k=60):
    scores = {}
    for ranks in ranks_list:
        for i, doc_id in enumerate(ranks):
            score = 1.0 / (k + i + 1)
            scores[doc_id] = scores.get(doc_id, 0) + score
    return sorted(scores.keys(), key=lambda x: -scores[x])
    • メリット：
      • テキスト・画像それぞれのスコアの絶対値に依存しない
      • 「どちらが上位に来ているか」という相対的な順位だけを見るため、埋め込み品質の差に強い

    重み付きスコア統合

    • テキストと画像の重要度をドメインごとに調整します。
    # 例：テキスト重み0.7、画像重み0.3
final_score = 0.7 * text_score + 0.3 * image_score
    • クエリの種類に応じて重みを変える（例：
      • 「図を探して」→ 画像重みを高く
      • 「説明文を探して」→ テキスト重みを高く
        ）といったクエリタイプ別の重み付けも有効ですAzure AI Search Docs。

    4. 対応策③：マルチモーダル埋め込みの統一（Joint Embedding）

    CLIP系の統一埋め込み

    • CLIPはテキストと画像を同じベクトル空間に埋め込むため、

      • テキストクエリ ↔ 画像ドキュメント
      • 画像クエリ ↔ テキストドキュメント
        を同じ距離尺度で比較できますOpenAI CLIP Paper。

    • テキストドキュメントもCLIPのテキストエンコーダで埋め込むことで、
      テキスト–テキスト検索もCLIP空間で統一できます。

    専用のマルチモーダル埋め込みモデル

    • 近年は、テキスト・画像・表などを統一的に扱うマルチモーダル埋め込みモデルも登場していますLangChain Blog。
    • 例：
      • 画像＋テキストを同時入力して一つの埋め込みを出力するモデル
      • 表形式データも含めて埋め込むモデル

    これらを使うと、「テキストだけ」「画像だけ」という分断を減らし、距離の不均一を緩和できます。

    5. 対応策④：リランキングで「最終順位」を補正

    クロスエンコーダ型リランキング

    • テキスト–テキスト、テキスト–画像の上位候補を対象に、
      • クエリとドキュメント（テキスト or 画像キャプション）を同時に入力するクロスエンコーダで再スコアリング
      • テキスト・画像の違いをモデル側で吸収してもらう
        というアプローチですSentence Transformers Docs。
    from sentence_transformers import CrossEncoder

model = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

pairs = [(query, doc_text) for doc_text in candidate_texts]
scores = model.predict(pairs)
    • メリット：
      • 埋め込みの「距離感の違い」を後段で補正できる
      • テキスト・画像どちらもテキスト化してから比較するため、モダリティ差を吸収しやすい

    LLMベースのリランキング

    • GPT-4などに
      • クエリ
      • 候補ドキュメント（テキスト＋画像キャプション）
        を渡し、「最も関連性の高い順に並べ替えて」と指示する方法もあります。
    • LLMは意味的な関連性を判断できるため、
      埋め込みの距離差による偏りを高レベルで補正できますPinecone。

    6. 対応策⑤：ドメイン適応・ファインチューニング

    埋め込みモデルのファインチューニング

    • 自社ドメインのテキスト＋画像ペアデータを使って、
      CLIPやテキスト埋め込みモデルを追加学習します。
    • これにより、
      • テキスト–画像の対応関係がドメイン特化に改善
      • テキスト–テキスト、テキスト–画像の距離分布が近づく
        効果が期待できますHugging Face Blog。

    コントラスト学習（Contrastive Learning）

    • 「同じ意味を持つテキスト–画像ペア」を近くに、
      「異なる意味のペア」を遠くに配置するように学習することで、
      テキストと画像の埋め込み空間をより整合的にできます。

    7. 評価とチューニングのポイント

    • 評価指標の設計

      • テキスト検索精度（nDCG@k）
      • 画像検索精度（Recall@k）
      • マルチモーダル検索全体の精度
        を別々に計測し、どのモダリティが弱いかを把握しますMeilisearch Blog。

    • A/Bテストでの重み調整

      • テキスト重み vs 画像重み
      • RRFのパラメータ（k値）
      • リランキングの有無
        を変えて、ユーザーフィードバックやクリック率で最適な設定を探ります。

    課題3: 埋め込みモデルのファインチューニングが結構難しい

    RAGで使う埋め込みモデルのファインチューニングは、「なぜ難しいのか」が意外と整理されていないことが多いです。主な理由を、技術面・データ面・評価面に分けて説明します。

    1. 技術的な難しさ

    (1) 埋め込みモデルは「中間表現」であり、直接の損失が定義しづらい

    • 分類モデルや生成モデル（LLM）は、
      • 「正解ラベルとのクロスエントロピー」
      • 「生成テキストとの損失」 といった明確な損失関数があります。
    • 一方、埋め込みモデルは
      • 「意味的に近いものを近く、遠いものを遠く」という相対的な距離を学習するため、
      • 「どの程度近ければよいか」という絶対的な正解がありません。
    • そのため、目的関数（損失）の設計が難しいことが多いですSentence Transformers Docs。

    (2) コントラスト学習の設計が難しい

    • 埋め込みのファインチューニングでは、コントラスト学習（Contrastive Learning） がよく使われますが、
      • 「正例（似ているペア）」「負例（似ていないペア）」の選び方
      • 負例の数・質（Hard Negative Mining） が性能に大きく影響しますHugging Face Blog。
    • 負例をランダムに選ぶと「簡単すぎる負例」ばかりになり、
      モデルがほとんど学習しないことがあります。

    (3) モデルサイズと計算コスト

    • 高品質な埋め込みモデル（例：text-embedding-3-large相当）はパラメータ数が多く、
      • ファインチューニングには大量のGPUメモリと計算時間が必要です。
    • 一方で、RAG用途では低レイテンシが求められるため、
      モデルを軽量化しようとすると表現力が落ちるというトレードオフがあります。

    2. データ面の難しさ

    (1) 高品質な「類似ペア」データの不足

    • RAG用の埋め込みをファインチューニングするには、
      • 「このクエリに対して、この文書が正解」
      • 「この文書とこの文書は意味的に近い」 といったラベル付きデータが必要です。
    • しかし、実際の業務データでは
      • クエリログはあっても「どの文書を参照したか」が記録されていない
      • 人手でアノテーションするコストが高い
        ため、十分な量の高品質データを用意するのが難しいですPinecone。

    (2) ドメイン特化データの偏り

    • 特定ドメイン（医療、法律、製造業など）では、
      • 専門用語
      • 略語
      • 文脈依存の意味
        が多く、汎用モデルでは捉えきれないことがあります。
    • しかし、そのドメインのラベル付きデータが少ないため、
      ファインチューニングしても過学習しやすいという問題があります。

    (3) クエリ分布の変化

    • RAGのクエリ分布は、時間とともに変化します（新製品、新規制、トレンドなど）。
    • 一度ファインチューニングしたモデルが、数ヶ月後には古くなる可能性があり、
      継続的な再学習が必要になりますが、その運用コストが高いです。

    3. 評価・検証の難しさ

    (1) 評価指標と実務の乖離

    • 埋め込みモデルの評価では、
      • nDCG@k
      • Recall@k
      • MRR（Mean Reciprocal Rank） などが使われますが、
        「RAG全体としての回答品質」と直接結びつかないことがありますMeilisearch Blog。
    • 埋め込みの精度が上がっても、
      • LLMがうまく情報を統合できない
      • コンテキスト長の制限で重要な情報が落ちる
        といった他の要因で最終回答の品質が変わらないこともあります。

    (2) オフライン評価とオンライン評価のギャップ

    • オフラインでは「テストセットに対するnDCG@10が改善した」としても、
      • 実際のユーザークエリでは改善が見られない
      • 特定のクエリタイプで悪化する
        といったことが起こりがちです。
    • オンライン評価（A/Bテスト）には
      • トラフィック分割
      • ログ収集
      • 分析
        の仕組みが必要で、運用コストが高いです。

    4. モデル選択・ハイパーパラメータの難しさ

    (1) どのモデルをファインチューニングするか

    • ベースモデルとして
      • 汎用モデル（OpenAI系、Sentence-BERT系）
      • ドメイン特化モデル（法律・医療向けなど） のどちらを選ぶかで、ファインチューニングの効果が大きく変わります。
    • 汎用モデルは表現力が高いがドメイン適応にデータが必要、
      ドメイン特化モデルは初期性能は高いが追加学習の余地が小さいことがあります。

    (2) ハイパーパラメータの感度

    • 学習率、バッチサイズ、エポック数、負例の数など、
      ハイパーパラメータの影響が大きいです。
    • 特にコントラスト学習では、
      • 学習率が高すぎると発散
      • 低すぎると収束が遅い
        といった問題があり、チューニングに時間がかかりますSentence Transformers Docs。

    5. 代替手段との比較（「やらない」選択肢の魅力）

    (1) リランキングの方が簡単で効果的

    • 埋め込みモデルをファインチューニングする代わりに、
      • 汎用埋め込みで広く検索
      • クロスエンコーダやLLMでリランキング
        という二段階パイプラインにすると、
      • 実装が簡単
      • 追加学習なしで精度向上 が期待できますPinecone。

    (2) プロンプトエンジニアリング・コンテキスト設計の方がコスパが良い

    • RAG全体の精度は、
      • 埋め込み
      • リランキング
      • LLMのプロンプト設計
      • コンテキストの切り出し方（チャンキング）
        など、複数の要因に依存します。
    • 埋め込みのファインチューニングにリソースを投じるより、
      プロンプトやチャンキング戦略の改善の方が、少ないコストで大きな効果が出ることも多いです。

    総括

    本日記事の要点をまとめると以下の通りです。

    1. 基本アプローチ

      • CLIPなどでテキストと画像を同じ空間に埋め込む
      • 画像をテキスト化（キャプション）してテキストRAGに流す
      • テキストと画像を別々に埋め込んでスコアを統合（RRFなど）
      • Azure AI Searchなどクラウドのマルチモーダル検索機能を使う

    2. 主な課題と対策

      • 複数物体が映る画像：
        • 画像を分割して埋め込む（グリッド・物体検出）
        • 対象ごとにキャプション生成してテキストRAG化
      • 画像とテキストの埋め込みの不均一：
        • 正規化・スケーリングで距離感を揃える
        • RRFや重み付き統合でスコアを統合
        • CLIPで統一埋め込みし、リランキングで順位を補正
      • 埋め込みのファインチューニングが難しい：
        • 損失設計・データ不足・評価の乖離・運用コストが高い
        • 実務ではリランキングやプロンプト改善の方がコスパが良いことが多い

    3. 実務の指針

      • まずは汎用埋め込み＋リランキング＋プロンプト改善で試す
      • 必要に応じて画像分割・キャプション生成・マルチベクトル検索を追加
      • 本格的なファインチューニングは評価・再学習体制が整ってから検討
