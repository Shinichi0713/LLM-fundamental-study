
RAGを行う場合、技術資料に掲載されている画像と関連する文章を両方参照したい(マルチモーダルRAGと呼ばれます)ということがあります。
それも結構な頻度で。

本日テーマ：
>画像を含めてRAGを実現する方法について実現法を整理します

## 概要

文章と画像を**一緒に検索したい**場合、代表的なアプローチは大きく次の4つに分けられます。


### 1. マルチモーダル埋め込みモデルを使う（CLIP系）

**CLIP**（Contrastive Language-Image Pretraining）のような**マルチモーダル埋め込みモデル**を使う方法です。

- テキストと画像を**同じベクトル空間**に埋め込む
- テキストクエリで画像を検索したり、画像クエリでテキストを検索したりできる
- 例：OpenAI CLIP、JinaCLIP、Apple DFN5B-H、Meta系モデルなど[Pinecone](https://www.pinecone.io/learn/series/image-search/clip)[Towards Data Science](https://towardsdatascience.com/clip-model-overview-unlocking-the-power-of-multimodal-ai)

**RAGへの組み込みイメージ**
- 文書中のテキストと画像をCLIPで埋め込み、同じベクトルDBに保存
- ユーザーのテキストクエリ → テキスト埋め込み → ベクトル検索 → テキスト・画像の両方を取得
- 取得したテキスト＋画像をマルチモーダルLLM（GPT-4Vなど）に渡して回答生成

**メリット**
- テキストと画像を**同じ空間で比較**できるため、セマンティックな検索が可能
- 画像そのものをベクトルとして扱える

**デメリット**
- モデルやインフラの準備が必要（CLIPのホスト、ベクトルDBなど）

### 2. 画像をテキスト化してRAGに組み込む

画像を**テキストで記述**し、それをRAGの文書として扱う方法です。

- GPT-4Vなどの**画像理解モデル**で画像を詳細に説明（キャプション生成）
- そのテキストを通常のテキストRAGパイプラインに流す
- 検索時は「テキストクエリ vs 画像キャプション」で類似度を計算

**RAGへの組み込みイメージ**
- 画像 → GPT-4Vで説明文生成 → テキストとしてベクトル化・保存
- 検索時はテキストクエリでキャプションを検索し、対応する画像も一緒に取得
- LLMに「テキスト＋画像URL」を渡して回答生成

**メリット**
- 既存のテキストRAGパイプラインをほぼそのまま流用できる
- 画像専用の埋め込みモデルが不要

**デメリット**
- 画像の細かい視覚情報がテキスト化で失われる可能性
- GPT-4VのAPIコストがかかる

### 3. ハイブリッド検索（テキスト＋画像のマルチベクトル）

**マルチベクトル・リトリーバー**のように、テキストと画像を**別々に埋め込み**、検索時に統合する方法です。

- テキスト用の埋め込み（例：OpenAI text-embedding-3-small）
- 画像用の埋め込み（例：CLIP）
- それぞれをベクトルDBに保存し、検索時に**スコアを統合**（例：Reciprocal Rank Fusion）

LangChainの「Multi-Vector Retriever」などがこの考え方に近いです[LangChain Blog](https://www.langchain.com/blog/semi-structured-multi-modal-rag)。

**メリット**
- テキストと画像で**最適なモデルを選べる**
- スコアの重み付けを調整しやすい

**デメリット**
- 実装がやや複雑（複数の埋め込み・インデックス管理）

### 4. クラウドサービスのマルチモーダル検索機能を使う

**Azure AI Search**や**OpenSearch**など、マルチモーダル検索をネイティブにサポートするサービスを使う方法です。

- Azure AI Search：テキスト＋画像を含むドキュメントをインポートし、マルチモーダル埋め込みで検索[Microsoft Learn](https://learn.microsoft.com/en-us/azure/search/multimodal-search-overview)
- OpenSearch：Titan Multimodal Embeddingsなどを使い、テキスト・画像・（テキスト＋画像）クエリをサポート[OpenSearch Blog](https://opensearch.org/blog/multimodal-semantic-search)

**メリット**
- インフラ構築が比較的簡単
- スケーラビリティや運用面の負担が少ない

**デメリット**
- サービス固有の制約（料金、モデル選択の自由度など）

## 課題1: 複数物体が映る画像の扱い

CLIP系のマルチモーダル埋め込みは「画像全体」を一つのベクトルにまとめるため、**画像内に複数の対象が写っていると、特定の対象だけを狙った検索が難しくなる**という問題があります。

### なぜCLIPだけでは弱いのか

- CLIPは**画像全体を一つの埋め込みベクトル**に圧縮します。
- そのため、「画像内のどの部分がクエリに対応しているか」を細かく制御できません。
- 複数の対象が写っている画像では、
  - 「犬と猫が写っている画像」に対して「犬だけ」を検索したい
  - 「複数の商品が写っている画像」に対して「特定の商品だけ」を検索したい
  といった**部分的なマッチング**が苦手になります。

### 対応策：大きく分けて3つのアプローチ

__1. 画像を分割して埋め込む（Region-level / Patch-level embedding）__

**画像を領域ごとに分割し、それぞれを別々に埋め込む**方法です。

- 例：
  - グリッド分割（例：3×3や4×4のパッチ）
  - 物体検出（YOLOなど）で**バウンディングボックスごとに切り出し**
- 各パッチ／領域をCLIPで埋め込み、**複数のベクトルとして保存**します。
- 検索時は、クエリと**各パッチのベクトル**を比較し、最もスコアの高いパッチに対応する画像をヒットさせます。

**メリット**
- 「画像全体」ではなく「画像の一部」を対象にした検索が可能
- 物体検出を使えば、**意味のある領域単位**で検索できる

**デメリット**
- ベクトル数が増えるため、ストレージ・検索コストが増加
- どのように分割するか（グリッドサイズ・物体検出の閾値など）のチューニングが必要

__2. マルチモーダルLLMで「対象ごとのキャプション」を生成する__

**GPT-4VなどのマルチモーダルLLM**を使って、

- 「この画像に写っている各対象を、それぞれ詳しく説明して」
- 「商品A、商品B、商品Cごとにキャプションを生成して」

といった指示で、**対象ごとのテキスト説明**を生成します。

その後、

- 各キャプションをテキスト埋め込み（例：OpenAI text-embedding-3-small）でベクトル化
- 検索時はクエリと**各キャプションのベクトル**を比較

という形で、**テキストベースのRAG**として扱います。

**メリット**
- 対象ごとの「意味的な説明」をテキストで持てるため、検索精度が上がりやすい
- 既存のテキストRAGパイプラインをほぼそのまま使える

**デメリット**
- GPT-4VなどのAPIコストがかかる
- 画像の細かい視覚情報がテキスト化で失われる可能性

__3. マルチベクトル検索＋リランキング__

**複数の埋め込み（画像全体・領域ごと・キャプションなど）を組み合わせる**方法です。

- 例：
  - 画像全体のCLIP埋め込み
  - 物体ごとのCLIP埋め込み
  - 各物体のテキストキャプションの埋め込み
- これらを**別々のベクトルとして保存**し、検索時に
  - 各ベクトルに対してスコアを計算
  - Reciprocal Rank Fusion（RRF）などで**スコアを統合**
  - 必要に応じて**リランキング（クロスエンコーダなど）**で順位を調整

**メリット**
- 「画像全体の文脈」と「特定の対象」の両方を考慮できる
- リランキングを組み合わせることで、より高精度な順位付けが可能

**デメリット**
- 実装がやや複雑（複数の埋め込み・インデックス管理）

## 課題2: 画像と文章の埋め込み表現の不均一

画像とテキストの埋め込みは、モデルや学習データが異なるため、**同じベクトル空間でも「距離感」や「品質」が不均一**になりがちです。この問題への対応策を、実務で使えるレベルまで整理します。

### 1. なぜ「距離・品質の不均一」が起きるのか

- **学習タスクの違い**  
  テキスト埋め込みは「文書検索・意味類似度」を主目的に学習されることが多く、  
  画像埋め込み（CLIPなど）は「画像–テキストの対応付け」を目的に学習されます。
- **スケール・分布の違い**  
  テキスト埋め込みと画像埋め込みで、  
  - ベクトルのノルム分布
  - 類似度スコアのレンジ  
  が異なるため、**同じ距離尺度で比較すると片方に偏る**ことがあります。
- **モデル間の性能差**  
  テキスト用の最新モデル（例：OpenAI text-embedding-3-large）と、  
  画像用のモデル（例：CLIP ViT-B/32）で**性能差が大きい**と、片方の埋め込みが「支配的」になり、検索結果が偏ります。

### 2. 対応策①：正規化とスケーリングで「距離感」を揃える

__ベクトルの正規化（Normalization）__

- テキスト・画像の埋め込みを**L2正規化**して、  
  - すべてのベクトルを単位球面上に配置
  - 内積＝コサイン類似度として扱う  
  ことで、**ノルムの違いによる偏りを減らせます**[OpenAI Cookbook](https://cookbook.openai.com/examples/embedding_comparisons)。

```python
# 例：OpenAIのtext-embedding-3-small / CLIP埋め込みを正規化
import numpy as np

def normalize(emb):
    norm = np.linalg.norm(emb)
    return emb / norm if norm > 0 else emb

text_emb_norm = normalize(text_emb)
image_emb_norm = normalize(image_emb)
```

__スコアのスケーリング（Score Scaling）__

- テキスト–テキスト、テキスト–画像の**類似度スコアの分布**を事前に計測し、  
  - 平均・分散を揃えるように線形変換（例：z-score scaling）
  - あるいはmin–maxスケーリング  
  を行い、**スコアのレンジを揃えます**。

```python
# 例：テキスト–テキスト / テキスト–画像スコアの分布を揃える
# μ_tt, σ_tt: テキスト–テキストスコアの平均・標準偏差
# μ_ti, σ_ti: テキスト–画像スコアの平均・標準偏差

def scale_score(score, source_mean, source_std, target_mean, target_std):
    z = (score - source_mean) / source_std
    return z * target_std + target_mean

# テキスト–画像スコアをテキスト–テキストの分布に合わせる
scaled_score = scale_score(
    raw_ti_score, μ_ti, σ_ti, μ_tt, σ_tt
)
```

### 3. 対応策②：マルチベクトル検索＋スコア統合

__Reciprocal Rank Fusion（RRF）__

- テキスト検索結果と画像検索結果を**別々にランキング**し、  
  RRFで**順位を統合**します[Elasticsearch Docs](https://www.elastic.co/guide/en/elasticsearch/reference/current/rrf.html)。

```python
# RRFの簡易実装例
def rrf(ranks_list, k=60):
    scores = {}
    for ranks in ranks_list:
        for i, doc_id in enumerate(ranks):
            score = 1.0 / (k + i + 1)
            scores[doc_id] = scores.get(doc_id, 0) + score
    return sorted(scores.keys(), key=lambda x: -scores[x])
```

- メリット：
  - テキスト・画像それぞれの**スコアの絶対値に依存しない**
  - 「どちらが上位に来ているか」という**相対的な順位**だけを見るため、埋め込み品質の差に強い

__重み付きスコア統合__

- テキストと画像の**重要度をドメインごとに調整**します。

```python
# 例：テキスト重み0.7、画像重み0.3
final_score = 0.7 * text_score + 0.3 * image_score
```

- クエリの種類に応じて重みを変える（例：  
  - 「図を探して」→ 画像重みを高く
  - 「説明文を探して」→ テキスト重みを高く  
  ）といった**クエリタイプ別の重み付け**も有効です[Azure AI Search Docs](https://learn.microsoft.com/en-us/azure/search/multimodal-search-overview)。

### 4. 対応策③：マルチモーダル埋め込みの統一（Joint Embedding）

__CLIP系の統一埋め込み__

- CLIPは**テキストと画像を同じベクトル空間に埋め込む**ため、  
  - テキストクエリ ↔ 画像ドキュメント
  - 画像クエリ ↔ テキストドキュメント  
  を**同じ距離尺度で比較**できます[OpenAI CLIP Paper](https://arxiv.org/abs/2103.00020)。

- テキストドキュメントもCLIPのテキストエンコーダで埋め込むことで、  
  **テキスト–テキスト検索もCLIP空間で統一**できます。

__専用のマルチモーダル埋め込みモデル__

- 近年は、**テキスト・画像・表などを統一的に扱うマルチモーダル埋め込みモデル**も登場しています[LangChain Blog](https://www.langchain.com/blog/semi-structured-multi-modal-rag)。
- 例：
  - 画像＋テキストを同時入力して一つの埋め込みを出力するモデル
  - 表形式データも含めて埋め込むモデル

これらを使うと、**「テキストだけ」「画像だけ」という分断を減らし、距離の不均一を緩和**できます。

### 5. 対応策④：リランキングで「最終順位」を補正

__クロスエンコーダ型リランキング__

- テキスト–テキスト、テキスト–画像の**上位候補**を対象に、  
  - クエリとドキュメント（テキスト or 画像キャプション）を**同時に入力**するクロスエンコーダで再スコアリング
  - テキスト・画像の違いを**モデル側で吸収**してもらう  
  というアプローチです[Sentence Transformers Docs](https://www.sbert.net/examples/applications/cross-encoder/README.html)。

```python
from sentence_transformers import CrossEncoder

model = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

pairs = [(query, doc_text) for doc_text in candidate_texts]
scores = model.predict(pairs)
```

- メリット：
  - 埋め込みの「距離感の違い」を**後段で補正**できる
  - テキスト・画像どちらも**テキスト化してから比較**するため、モダリティ差を吸収しやすい

__LLMベースのリランキング__

- GPT-4などに  
  - クエリ
  - 候補ドキュメント（テキスト＋画像キャプション）  
  を渡し、「最も関連性の高い順に並べ替えて」と指示する方法もあります。
- LLMは**意味的な関連性**を判断できるため、  
  埋め込みの距離差による偏りを**高レベルで補正**できます[Pinecone](https://www.pinecone.io/learn/series/rag/rerankers)。

### 6. 対応策⑤：ドメイン適応・ファインチューニング

__埋め込みモデルのファインチューニング__

- 自社ドメインの**テキスト＋画像ペアデータ**を使って、  
  CLIPやテキスト埋め込みモデルを**追加学習**します。
- これにより、
  - テキスト–画像の対応関係が**ドメイン特化**に改善
  - テキスト–テキスト、テキスト–画像の**距離分布が近づく**  
  効果が期待できます[Hugging Face Blog](https://huggingface.co/blog/fine-tune-clip)。

__コントラスト学習（Contrastive Learning）__

- 「同じ意味を持つテキスト–画像ペア」を近くに、  
  「異なる意味のペア」を遠くに配置するように学習することで、  
  **テキストと画像の埋め込み空間をより整合的に**できます。

### 7. 評価とチューニングのポイント

- **評価指標の設計**  
  - テキスト検索精度（nDCG@k）
  - 画像検索精度（Recall@k）
  - マルチモーダル検索全体の精度  
  を**別々に計測**し、どのモダリティが弱いかを把握します[Meilisearch Blog](https://www.meilisearch.com/blog/multimodal-rag)。

- **A/Bテストでの重み調整**  
  - テキスト重み vs 画像重み
  - RRFのパラメータ（k値）
  - リランキングの有無  
  を変えて、**ユーザーフィードバックやクリック率**で最適な設定を探ります。

## 課題3: 埋め込みモデルのファインチューニングが結構難しい

RAGで使う埋め込みモデルのファインチューニングは、**「なぜ難しいのか」が意外と整理されていない**ことが多いです。主な理由を、技術面・データ面・評価面に分けて説明します。

### 1. 技術的な難しさ

__(1) 埋め込みモデルは「中間表現」であり、直接の損失が定義しづらい__

- 分類モデルや生成モデル（LLM）は、
  - 「正解ラベルとのクロスエントロピー」
  - 「生成テキストとの損失」
  といった**明確な損失関数**があります。
- 一方、埋め込みモデルは
  - 「意味的に近いものを近く、遠いものを遠く」という**相対的な距離**を学習するため、
  - 「どの程度近ければよいか」という**絶対的な正解**がありません。
- そのため、**目的関数（損失）の設計が難しい**ことが多いです[Sentence Transformers Docs](https://www.sbert.net/docs/training/overview.html)。

__(2) コントラスト学習の設計が難しい__

- 埋め込みのファインチューニングでは、**コントラスト学習（Contrastive Learning）** がよく使われますが、
  - 「正例（似ているペア）」「負例（似ていないペア）」の選び方
  - 負例の数・質（Hard Negative Mining）
  が性能に大きく影響します[Hugging Face Blog](https://huggingface.co/blog/how-to-train-sentence-embeddings)。
- 負例をランダムに選ぶと「簡単すぎる負例」ばかりになり、  
  **モデルがほとんど学習しない**ことがあります。

__(3) モデルサイズと計算コスト__

- 高品質な埋め込みモデル（例：`text-embedding-3-large`相当）はパラメータ数が多く、
  - ファインチューニングには**大量のGPUメモリと計算時間**が必要です。
- 一方で、RAG用途では**低レイテンシ**が求められるため、  
  モデルを軽量化しようとすると**表現力が落ちる**というトレードオフがあります。

### 2. データ面の難しさ

__(1) 高品質な「類似ペア」データの不足__

- RAG用の埋め込みをファインチューニングするには、
  - 「このクエリに対して、この文書が正解」
  - 「この文書とこの文書は意味的に近い」
  といった**ラベル付きデータ**が必要です。
- しかし、実際の業務データでは
  - クエリログはあっても「どの文書を参照したか」が記録されていない
  - 人手でアノテーションするコストが高い  
  ため、**十分な量の高品質データを用意するのが難しい**です[Pinecone](https://www.pinecone.io/learn/series/rag/fine-tuning-embeddings)。

__(2) ドメイン特化データの偏り__

- 特定ドメイン（医療、法律、製造業など）では、
  - 専門用語
  - 略語
  - 文脈依存の意味  
  が多く、**汎用モデルでは捉えきれない**ことがあります。
- しかし、そのドメインの**ラベル付きデータが少ない**ため、  
  ファインチューニングしても**過学習しやすい**という問題があります。

__(3) クエリ分布の変化__

- RAGのクエリ分布は、時間とともに変化します（新製品、新規制、トレンドなど）。
- 一度ファインチューニングしたモデルが、**数ヶ月後には古くなる**可能性があり、  
  **継続的な再学習が必要**になりますが、その運用コストが高いです。

### 3. 評価・検証の難しさ

__(1) 評価指標と実務の乖離__

- 埋め込みモデルの評価では、
  - nDCG@k
  - Recall@k
  - MRR（Mean Reciprocal Rank）
  などが使われますが、  
  **「RAG全体としての回答品質」と直接結びつかない**ことがあります[Meilisearch Blog](https://www.meilisearch.com/blog/multimodal-rag)。
- 埋め込みの精度が上がっても、
  - LLMがうまく情報を統合できない
  - コンテキスト長の制限で重要な情報が落ちる  
  といった**他の要因で最終回答の品質が変わらない**こともあります。

__(2) オフライン評価とオンライン評価のギャップ__

- オフラインでは「テストセットに対するnDCG@10が改善した」としても、
  - 実際のユーザークエリでは改善が見られない
  - 特定のクエリタイプで悪化する  
  といったことが起こりがちです。
- オンライン評価（A/Bテスト）には
  - トラフィック分割
  - ログ収集
  - 分析  
  の仕組みが必要で、**運用コストが高い**です。

### 4. モデル選択・ハイパーパラメータの難しさ

__(1) どのモデルをファインチューニングするか__

- ベースモデルとして
  - 汎用モデル（OpenAI系、Sentence-BERT系）
  - ドメイン特化モデル（法律・医療向けなど）
  のどちらを選ぶかで、**ファインチューニングの効果が大きく変わります**。
- 汎用モデルは表現力が高いが**ドメイン適応にデータが必要**、  
  ドメイン特化モデルは初期性能は高いが**追加学習の余地が小さい**ことがあります。

__(2) ハイパーパラメータの感度__

- 学習率、バッチサイズ、エポック数、負例の数など、  
  **ハイパーパラメータの影響が大きい**です。
- 特にコントラスト学習では、
  - 学習率が高すぎると発散
  - 低すぎると収束が遅い  
  といった問題があり、**チューニングに時間がかかります**[Sentence Transformers Docs](https://www.sbert.net/docs/training/overview.html)。

### 5. 代替手段との比較（「やらない」選択肢の魅力）

__(1) リランキングの方が簡単で効果的__

- 埋め込みモデルをファインチューニングする代わりに、
  - 汎用埋め込みで広く検索
  - クロスエンコーダやLLMでリランキング  
  という**二段階パイプライン**にすると、
  - 実装が簡単
  - 追加学習なしで精度向上
  が期待できます[Pinecone](https://www.pinecone.io/learn/series/rag/rerankers)。

__(2) プロンプトエンジニアリング・コンテキスト設計の方がコスパが良い__

- RAG全体の精度は、
  - 埋め込み
  - リランキング
  - LLMのプロンプト設計
  - コンテキストの切り出し方（チャンキング）  
  など、**複数の要因に依存**します。
- 埋め込みのファインチューニングにリソースを投じるより、  
  **プロンプトやチャンキング戦略の改善の方が、少ないコストで大きな効果**が出ることも多いです。


## 総括

本日記事の要点をまとめると以下の通りです。

![1780435196743](image/7_chart_rag/1780435196743.png)

1. **基本アプローチ**
   - CLIPなどで**テキストと画像を同じ空間に埋め込む**  
   - 画像を**テキスト化（キャプション）してテキストRAGに流す**  
   - テキストと画像を**別々に埋め込んでスコアを統合（RRFなど）**  
   - Azure AI Searchなど**クラウドのマルチモーダル検索機能**を使う

2. **主な課題と対策**
   - **複数物体が映る画像**：  
     - 画像を**分割して埋め込む**（グリッド・物体検出）  
     - 対象ごとに**キャプション生成**してテキストRAG化  
   - **画像とテキストの埋め込みの不均一**：  
     - **正規化・スケーリング**で距離感を揃える  
     - **RRFや重み付き統合**でスコアを統合  
     - **CLIPで統一埋め込み**し、**リランキング**で順位を補正  
   - **埋め込みのファインチューニングが難しい**：  
     - 損失設計・データ不足・評価の乖離・運用コストが高い  
     - 実務では**リランキングやプロンプト改善の方がコスパが良い**ことが多い

3. **実務の指針**
   - まずは**汎用埋め込み＋リランキング＋プロンプト改善**で試す  
   - 必要に応じて**画像分割・キャプション生成・マルチベクトル検索**を追加  
   - 本格的なファインチューニングは**評価・再学習体制が整ってから**検討
