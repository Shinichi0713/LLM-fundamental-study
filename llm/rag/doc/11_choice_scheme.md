
GraphRAGにデータ登録するときに意識すべきものにトリプルという概念があります。
何を隠そう、トリプルの不理解で、登録したデータを、作った検索アルゴリズムでなかなかヒットされないという事態を招いていました。
本日はトリプルの概念について説明していきます。

## 概要

### トリプルを構成する3つの要素

トリプルは、必ず以下の順番で構成されます。

```text
[ 主語 (Subject) ] ───[ 述語 (Predicate) ]───> [ 目的語 (Object) ]
```

- **主語（Subject）**  
  情報の出発点となる「モノ」や「人」、「概念」です（グラフ構造における**ノード**）。
- **述語（Predicate）**  
  主語と目的語がどういう関係にあるかを表す「矢印」です。属性や関係性を定義します（グラフ構造における**エッジ / リンク**）。
- **目的語（Object）**  
  情報の行き先となる「モノ」や「人」、あるいは具体的な「値（テキストや数値）」です（グラフ構造における別の**ノード**、または**プロパティ値**）。


### 具体的な例

例えば、「**山田さんはA社に勤務している**」という1つの文章（データ）をトリプルに分解すると、以下のようになります。

| 要素 | 実際のデータ | グラフでの役割 |
| --- | --- | --- |
| **主語 (Subject)** | 山田さん | ノード（出発点） |
| **述語 (Predicate)** | 勤務している (WORKS_AT) | 矢印（関係性） |
| **目的語 (Object)** | A社 | ノード（到達点） |

これをLlamaIndexやLLMの設定プログラムに渡すと、システムは裏側で以下のような3つの塊（トリプル）として認識し、GraphDBに「点と線」として描き込みます。

> `("山田さん", "WORKS_AT", "A社")`

__値を表現するトリプル__

トリプルは「人と組織」のような関係性だけでなく、「年齢」や「設立日」といったステータス（属性）を表現する際にも使われます。

- `("山田さん", "年齢", "30")`
- `("A社", "設立年", "2015")`

__なぜトリプルという形にするのか？__

1つひとつのデータは「主語・述語・目的語」という単純な3要素ですが、これが大量に集まって**主語や目的語が連鎖的につながる**ことで、網の目のようなどこまでも拡張できる巨大な「知識グラフ」が完成します。

例えば、

- `("山田さん", "WORKS_AT", "A社")`
- `("A社", "DEVELOPED", "次世代AIツール")`
- `("次世代AIツール", "USES", "Python")`

この3つのトリプルがデータベースに登録されるだけで、システムは自動的に「山田さんはPythonを使ったツールの開発に関わっている会社にいるんだな」という、直接書かれていない複雑な文脈を芋づる式に検索（GraphRAGなど）できるようになります。

直接書かれていない文脈まで芋づる式に検索できるのは、トリプル構造によってデータが「独立したテーブル（表）」ではなく「1つの巨大なネットワーク（道路網）」としてデータベースに格納されるからです。

### トリプルで無限なネットワークとなる理由

なぜトリプルだと `網の目のようなどこまでも拡張できる巨大な「知識グラフ」` が可能になるのか、その理由は主に3つあります。

__1. 「目的語」が次のトリプルの「主語」に変身するから（連鎖性）__

トリプルの最大の特徴は、ある情報の「ゴール（目的語）」が、別の情報の「スタート（主語）」にシームレスに入れ替われる点にあります。

先ほどの例をもう一度見てみましょう。

* トリプル①：山田さん ──[WORKS_AT]──> **A社**
* トリプル②：**A社** ──[DEVELOPED]──> **次世代AIツール**
* トリプル③：**次世代AIツール** ──[USES]──> Python

データベース内では、トリプル①のゴールである `A社` と、トリプル②のスタートである `A社` が完全に「同一の点（ノード）」として統合されます。

システムから見ると、**「山田さん」から矢印を2回（WORKS_AT → DEVELOPED）たどるだけで、自動的に「次世代AIツール」にたどり着く一本の道**が最初から開通している状態になります。

__2. 「結合（JOIN）」という重い処理がいらないから__

従来の一般的なデータベース（関係系データベース：RDB）で同じことをやろうとすると、「社員テーブル」「会社テーブル」「製品テーブル」「技術テーブル」のように別々の表にデータを小分けにします。

これらを繋げて検索するには、検索するたびにコンピュータが表と表を突き合わせる「結合（JOIN）」という非常に重い計算処理を行う必要があります。

一方、トリプルを使ったGraphDBは、データを入れた時点で**最初からすべてのデータが線で結ばれた状態**で保存されています。システムは表を検索するのではなく、ただ「線を指でたどるだけ（ポインタの移動）」なので、何段階データが離れていても、一瞬で芋づる式に情報を引っ張り出せるのです。

__3. LLM（AI）が「文脈の近さ」を理解しやすいから__

最近のGraphRAGなどでこの仕組みが注目されているのは、LLMとの相性が抜群だからです。

「山田さんはPythonを使ったツールの開発に関わっている会社にいる」という結論を導くには、人間なら脳内で知識を繋ぎ合わせますよね。GraphDBは、まさに**人間の脳の記憶（ニューラルネットワーク）と同じ構造**でデータを保持しています。

LLMに「山田さんに関連する情報を探して」と頼むと、LlamaIndexなどのシステムは「山田さん」のノードから2歩、3歩先までのトリプル（線）をまとめてごっそり取得し、LLMに渡します。
LLMはその「繋がった状態のデータ」を読み込むため、直接文章として書かれていなくても、「なるほど、山田さん─A社─ツール─Pythonという関係があるんだな」と正確に文脈を理解して回答できるようになります。


**一言で言えば：**
バラバラの箇条書きの知識ではなく、最初から「全ての知識が数珠繋ぎになった地図」としてDBに保管されているため、システムが迷子にならずに隠れた関係性を見つけ出せるのです。


## LlamaIndexでのスキーマ設計の違い

### 1. スキーマレスな抽出：`SimpleLLMPathExtractor`

データ登録にLLamaIndexの `SimpleLLMPathExtractor` を使っているため、**スキーマ（エンティティ種別や関係種別）は明示的に定義されていません**。  
この抽出器はデフォルトのプロンプトテンプレートで各チャンクをLLMに渡し、

```text
(entity1, relation, entity2)
```

という形式のトリプルを**LLMの判断に任せて自由に**生成させます。そのため、

- エンティティのラベル（`Person`, `Organization` など）
- 関係の種類（`WORKS_FOR`, `LOCATED_IN` など）

はすべて **LLMが文脈から適当に命名**します。実行するたびに表記ゆれ（例: `founded` と `Founded` と `is_founder_of` が混在する）が起きやすいのが弱点です。

Neo4j側では、各トリプルは概ね以下のような形で格納されます。

```cypher
(:Entity {name: "..."}) -[:RELATION_NAME {...}]-> (:Entity {name: "..."})
```

`Entity` という汎用ラベルにプロパティとして名前が入り、リレーション名がLLMの出力そのまま使われる、というイメージです。

### 2. スキーマを固定したい場合：`SchemaLLMPathExtractor`

エンティティ種別・関係種別・その組み合わせを事前に定義し、LLMにその枠内で抽出させたい場合は `SchemaLLMPathExtractor` を使います。

```python
from llama_index.core.indices.property_graph import SchemaLLMPathExtractor
from typing import Literal

# エンティティの種別を定義
entities = Literal["PERSON", "ORGANIZATION", "LOCATION", "EVENT", "PRODUCT"]

# 関係の種別を定義
relations = Literal["WORKS_FOR", "FOUNDED", "LOCATED_IN", "PARTICIPATED_IN", "PRODUCES"]

# (エンティティ, 関係, エンティティ) の許可される組み合わせ(トリプル)を定義
validation_schema = {
    "PERSON": ["WORKS_FOR", "FOUNDED", "PARTICIPATED_IN"],
    "ORGANIZATION": ["LOCATED_IN", "PRODUCES"],
    "EVENT": ["LOCATED_IN"],
}

kg_extractor = SchemaLLMPathExtractor(
    llm=Settings.llm,
    possible_entities=entities,
    possible_relations=relations,
    kg_validation_schema=validation_schema,
    strict=True,  # True: スキーマ外の抽出は除外 / False: 参考程度に留める
)
```

この `kg_extractor` を `PropertyGraphIndex.from_documents(..., kg_extractors=[kg_extractor], ...)` の部分に差し替えるだけで、Neo4j側には一貫したラベル・関係名でノードが登録されるようになります。

### 3. Neo4j自体のスキーマ設定について

Neo4jはスキーマレスDBなので、テーブル定義のようなものはありません。ただし運用上は以下のような設定を別途行うと良いです。

- **制約・インデックス**: 重複ノード防止や検索高速化のために `CREATE CONSTRAINT` や `CREATE INDEX` をCypherで発行しておく（LlamaIndexは自動では作りません）
- **ベクトルインデックス**: `embed_kg_nodes=True` にした場合、`Neo4jPropertyGraphStore` はデフォルトでベクトル検索用のインデックスを内部的に作成します

```cypher
CREATE CONSTRAINT entity_name IF NOT EXISTS
FOR (e:Entity) REQUIRE e.name IS UNIQUE;
```

### 4. 抽出器の比較

| 抽出器 | スキーマ | 特徴 |
|---|---|---|
| `SimpleLLMPathExtractor`（元コード） | なし（自由生成） | 手軽だが表記ゆれが出やすい |
| `SchemaLLMPathExtractor` | あり（entities/relations/triple定義） | 一貫性が高く、後続のクエリ・可視化がしやすい |



LlamaIndexがNeo4j（GraphDB）を検索する仕組みは、従来のテキストの「見た目の似ている度」だけで探す検索（ベクトル検索）とは全く異なります。

LlamaIndexの `PropertyGraphIndex` では、ユーザーからの質問に対して、主に3つの異なる検索アプローチをハイブリッド（統合）させて答えを導き出しています。

---

## 1. ベクトル＋サブグラフ探索（Vector + Subgraph）

これが最も標準的かつ強力な検索手法です。

* **ステップ A**: ユーザーが「山田さんの現在のプロジェクトは？」と質問します。
* **ステップ B**: LlamaIndexはまず、質問文のベクトル（意味）に最も近いノード（例：「山田さん」）をグラフ内から見つけ出します。
* **ステップ C**: 次に、その見つかった「山田さん」ノードから**矢印（エッジ）を1〜2歩たどって、繋がっている周囲のトリプルをまるごと（サブグラフとして）取得**します。
> 例：`(山田さん)-[WORKS_AT]->(A社)-[MANAGES]->(AIプロジェクト)`


* **ステップ D**: 抽出された「繋がった状態のデータ（文脈）」がLLMに渡され、LLMが綺麗な日本語の回答を作ります。

## 2. Text-to-Cypher（クエリ自動生成）

ユーザーの自然言語を、LLMを使ってGraphDBの専用クエリ言語である「Cypher（サイファー）」に変換して検索する仕組みです。

* **ステップ A**: ユーザーが「Pythonを使っている社内の人間を全員リストアップして」と質問します。
* **ステップ B**: LlamaIndexの裏側にいるLLMが、登録されているスキーマ（設計図）を見ながら、自動で以下のようなCypherクエリを組み立てます。
```cypher
MATCH (p:PERSON)-[:WORKS_AT]->()-[:USES]->(t:TECHNOLOGY {name: 'Python'}) 
RETURN p.name

```


* **ステップ C**: このクエリをNeo4jに直接実行し、データベースからピンポイントで正確なデータを構造化された状態で一瞬で取得します。集計や条件の絞り込みに非常に強い特徴があります。

## 3. キーワード / シノニム（同義語）検索

言葉のゆらぎを補正するための検索です。

* **ステップ A**: ユーザーが「AI事業の担当者は？」と質問します。
* **ステップ B**: グラフ内に「AI事業」という一言が直接なくても、LLMが「AI事業 ＝ 機械学習プロジェクト、次世代AIツール」と同義語（シノニム）を連想します。
* **ステップ C**: 連想されたキーワードに部分一致するノードを起点として、関連するトリプルを芋づる式に手繰り寄せます。

---

## 💡 最大の強み：これらを「いいとこ取り」する

LlamaIndexが優秀なのは、どれか1つを使うのではなく、`index.as_query_engine()` を動かしたときに、これらの検索結果をすべて集めてブレンドする（Reciprocal Rank Fusionなどを用いて順位付けする）点にあります。

* **ベクトル検索**で「文脈のニュアンス」を捉え、
* **グラフ（トリプル）構造**で「正確な事実の繋がり」を担保し、
* **Cypher**で「複雑な条件」を処理する。

この3層構造の連携によって、一般的なRAG（社内文書検索）で頻発する「情報は引っ張れているけど、Aさんの話とB社の話がごっちゃになって嘘の回答（ハルシネーション）を作る」という問題を劇的に減らすことができるのが、LlamaIndex×Neo4jの検索の正体です。
