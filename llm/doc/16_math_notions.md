## 良く使われる数理モデル
LLM（大規模言語モデル）の数理モデルは、基本的に **Transformer アーキテクチャ** をベースにしているので、その論文で使われる表記がほぼ共通の「標準」になっています。

代表的なものを整理すると、だいたい次のような表記が主流です。


### 1. 入力・埋め込みまわりの表記

- **トークン列の長さ**：  
  - $ n $ や $ L $（sequence length）
- **埋め込み次元**：  
  - $ d_{\text{model}} $ あるいは単に $ d $（hidden dimension）
- **入力行列**：  
  - $ X \in \mathbb{R}^{n \times d} $ や $ \mathbf{X} \in \mathbb{R}^{n \times d} $（トークン数×次元の行列）

### 2. Attention（Q, K, V）まわりの表記

- **Query, Key, Value**：  
  - $ Q, K, V $（それぞれ Query, Key, Value の頭文字）
  - あるいは $ \mathbf{Q}, \mathbf{K}, \mathbf{V} $（太字で行列を表す）
- **Attention 重み（スコア）**：  
  - $ \text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^\top}{\sqrt{d_k}}\right)V $
  - ここで $ d_k $ は Key の次元（head dimension）
- **マルチヘッド Attention**：  
  - $ \text{MultiHead}(Q, K, V) = \text{Concat}(\text{head}_1, \dots, \text{head}_h)W^O $
  - $ \text{head}_i = \text{Attention}(QW_i^Q, KW_i^K, VW_i^V) $
  - $ h $ はヘッド数、$ W_i^Q, W_i^K, W_i^V, W^O $ は学習される重み行列


### 3. 正規化・残差接続まわりの表記

- **Layer Normalization**：  
  - $ \text{LayerNorm}(x) $ あるいは $ \text{LN}(x) $
  - 最近の LLM では RMSNorm もよく使われ、$ \text{RMSNorm}(x) $ と書かれることが多い
- **残差接続**：  
  - $ x + \text{Sublayer}(x) $ あるいは $ x + F(x) $
  - Pre-Norm か Post-Norm かで順序が変わる

### 4. フィードフォワードネットワーク（FFN / MLP）

- **FFN の表記**：  
  - $ \text{FFN}(x) = W_2\,\sigma(W_1 x + b_1) + b_2 $
  - あるいは $ \text{MLP}(x) $ と書く
  - 最近の LLM では SwiGLU など Gated Linear Unit 系が多く、  
    $ \text{FFN}(x) = (W_2 \,\text{swish}(W_1 x)) \odot (W_3 x) $ のような形で書かれることも多い


### 5. 位置エンコーディング（Positional Encoding）

- **絶対位置エンコーディング（sin/cos）**：  
  - $ PE_{(pos, 2i)} = \sin(pos / 10000^{2i/d}) $
  - $ PE_{(pos, 2i+1)} = \cos(pos / 10000^{2i/d}) $
- **RoPE（Rotary Positional Embedding）**：  
  - 回転行列 $ R_{\Theta, m} $ を使って $ q_m = R_{\Theta, m} W_q x_m $ のように書かれることが多い

### 6. 出力・損失まわりの表記

- **出力確率分布**：  
  - $ P(w_t \mid w_{<t}) = \text{softmax}(W_o h_t) $
  - ここで $ h_t $ は時刻 $ t $ の隠れ状態、$ W_o $ は出力重み
- **損失（クロスエントロピー）**：  
  - $ \mathcal{L} = -\sum_t \log P(w_t \mid w_{<t}) $


### 7. その他のよく見る表記

- **層のインデックス**：  
  - $ \ell = 1, \dots, L $（層数）
- **ヘッドのインデックス**：  
  - $ i = 1, \dots, h $
- **トークン位置**：  
  - $ t $ や $ i, j $（$ i $ 番目のトークン、$ j $ 番目のトークン）

__数理モデル表記__

- **Q, K, V**（Query, Key, Value）
- **LayerNorm / RMSNorm**
- **softmax**（Attention の重み計算や出力確率）
- **FFN / MLP**（フィードフォワードネットワーク）
- **RoPE** などの位置エンコーディング
- **クロスエントロピー損失** $ \mathcal{L} $

といった表記が、LLM 論文の数理モデルではほぼ共通して使われています。  
細かい記号（添字の付け方や太字かどうか）は論文ごとに多少揺れがありますが、上記の「意味」と「略記」はかなり標準化されています。

## 著者の意図
著者も趣味や好きだから数理モデルを使っているわけではないと考えてます。
LLM論文の著者が数理モデルを使う理由は、大きく分けて次のようなものです。

### 1. モデルの構造を正確に定義するため

LLMは複雑なニューラルネットワークなので、**言葉だけでは構造を厳密に伝えられません**。

- 「Attention で各トークンが他のトークンを見る」と口で言っても、  
  どういう計算で「見る」のかは曖昧です。
- 数式で
  $$
  \text{Attention}(Q,K,V) = \text{softmax}\left(\frac{QK^\top}{\sqrt{d_k}}\right)V
  $$
  と書けば、**どの行列をどう掛けて、どこで softmax を取るか**が一意に決まります。

つまり、**「何をどう計算しているか」を正確に共有する**ために数理モデルが必要です。

### 2. 再現性（Reproducibility）を確保するため

研究コミュニティでは、他の研究者が同じ結果を再現できることが重要です。

- 数式でモデルを書いておけば、  
  誰でも同じ式を実装して同じ構造のモデルを構築できます。
- もし言葉だけで「こういう感じのAttentionです」と書くと、  
  解釈の幅が広すぎて、再現実験が難しくなります。

数理モデルは、**「この論文のモデルはこういう式で定義されている」という共通仕様**の役割を果たします。

### 3. 理論的な解析や証明の土台にするため

LLMの振る舞いを理論的に理解したいとき、数式がないと議論が進みません。

- 例えば「層を深くすると勾配がどうなるか」「Attention の表現力はどれくらいか」といった**理論解析**は、  
  数理モデルを前提に行われます。
- また、新しい正規化手法や Attention の変種を提案するとき、  
  「なぜこれがうまくいくのか」を**数学的に説明**する必要があります。

数理モデルは、**理論的な主張を支える共通言語**として機能します。


### 4. アルゴリズムや実装を明確に伝えるため

LLMの学習・推論アルゴリズムも、数式で書くと明確になります。

- 例えば「KVキャッシュを使った高速推論」を説明するとき、  
  どのベクトルをキャッシュし、どのタイミングで更新するかを数式で書くと、  
  実装者が迷わずコードに落とし込めます。
- また、バッチ処理や並列化の都合で、  
  どの次元をどう並べ替えるかといった**実装上の工夫**も、  
  数式で表現すると理解しやすくなります。

### 5. モデルのバリエーションや一般化を議論するため

「標準的なTransformer」から少し変えた新しいアーキテクチャを提案するとき、  
**どこをどう変えたか**を数式で示す必要があります。

- 例えば「QK-Norm を導入した」「RMSNorm を使った」「RoPE をこう変形した」などは、  
  数式で差分を書かないと、読者に伝わりません。
- また、一般化された形（例：任意のヘッド数・次元数）で書いておけば、  
  別の設定（モデルサイズやタスク）に応用しやすくなります。

### 6. コミュニティ内の共通言語として使うため

LLM研究コミュニティでは、  
- $Q, K, V$
- LayerNorm / RMSNorm
- FFN / MLP
- RoPE
- Cross-entropy loss

といった**記号や略語が共通語**になっています。

数理モデルを使うことで、  
- 論文同士の比較がしやすい
- 既存研究の式を引用しやすい
- 新しい提案が既存手法とどう違うかが明確になる

というメリットがあります。


### 7. 実装者（エンジニア）に設計意図を伝えるため

LLM論文は、研究者だけでなく、実際にモデルを実装するエンジニアも読者です。

- 数式を見れば、どの部分が計算のボトルネックになりそうか
- どの演算がメモリを多く使うか
- どの部分を最適化すべきか

といった**設計上の勘所**が読み取れます。

言葉だけの説明だと、こうした実装上のポイントがぼやけてしまいます。


## 理解する上でのコツ

LLMの数理モデル（Transformer系）を読んで理解するコツは、だいたい次のような順番で進めると良いのではと思っています。


### 1. まず「何をやっているか」を日本語でざっくり理解する

数式をいきなり追いかけるのではなく、**その式が何を実現しようとしているか**を先に押さえます。

- Attention なら「各トークンが他のトークンをどれだけ参照するかを決めて、その重み付き和を取る仕組み」
- LayerNorm なら「各トークンのベクトルを平均0・分散1付近に揃えて、学習を安定させる」
- FFN なら「各トークンごとに非線形変換をかけて表現を豊かにする」

というように、**「この式は何をやっているのか？」** を一言で言えるようにしておくと、後で数式を追いやすくなります。

### 2. 次元（shape）を常に意識する

LLMの数式は、行列やベクトルの演算がほとんどです。  
**どの次元が何を表しているか**を意識すると、理解が一気に楽になります。

- $ X \in \mathbb{R}^{n \times d} $：  
  - $ n $：トークン数（文の長さ）  
  - $ d $：隠れ次元（埋め込み次元）
- $ Q = X W_Q \in \mathbb{R}^{n \times d_k} $：  
  - $ d_k $：1ヘッドあたりの次元（head dimension）
- Attention のスコア行列：  
  - $ QK^\top \in \mathbb{R}^{n \times n} $（各行が「あるトークンが他の全トークンにどれだけ注目するか」）

といった具合に、**「この行列の行は何？列は何？」** を常に頭の中で確認しながら読むと、式の意味がはっきりします。

### 3. 小さい例（具体例）で手計算してみる

論文の式は一般形で書かれているので、**自分で小さな例を作って手計算**すると理解が深まります。

- トークン数 $ n = 3 $、次元 $ d = 4 $ くらいの小さな行列 $ X $ を適当に作る
- $ W_Q, W_K, W_V $ も小さな行列を仮定して、$ Q, K, V $ を計算してみる
- $ QK^\top $ を計算して、softmax をかけてみる
- その重みで $ V $ を加重平均してみる

実際に数値を入れてみると、「あ、これが Attention の重み行列なんだな」「この列が各トークンの出力になるんだな」と腑に落ちます。

### 4. 図（アーキテクチャ図）と数式を対応づける

多くの論文には Transformer のブロック図が載っています。

- 図の中で「ここが Q」「ここが K」「ここが V」と対応づける
- 図の「Add & Norm」が、数式では $ x + \text{LayerNorm}(F(x)) $ に対応する、といった具合に

**図を眺めながら数式を読む**と、どの式がどのブロックに対応しているかが分かりやすくなります。


### 5. 1ブロックだけに集中して理解する

LLM は同じブロックを何十層も積み重ねたものです。  
一度に全部を理解しようとせず、

- **1つの Transformer ブロック**（Self-Attention + FFN + LayerNorm + Residual）だけに絞る
- そのブロックの入出力の関係を完全に理解する
- それが分かれば、あとはそれを何層も積んだだけ、と考える

というように、**局所的な構造を徹底的に理解する**のがコツです。


### 6. 実装（PyTorch など）と見比べる
最近の論文は1/2くらいの確率で論文を試行したコードをgithubで公開しています。
論文の数式だけでは分かりにくい部分も、**公開されたコードを見ると一気に分かる**ことが多いです。

- Hugging Face の Transformers ライブラリや、PyTorch の Transformer 実装
- あるいは「Transformer from scratch」系の解説記事のコード

を見ながら、

- 論文の式のどの部分が、コードのどの行に対応しているか
- 次元の変換（view, permute, matmul）がどう行われているか

を確認すると、抽象的な数式が具体的な操作として理解できます。


### 7. 自分で図やメモを書く

読むだけだと頭に入りにくいので、

- 自分なりのブロック図を描く
- 式の変形をノートに書き写す
- 各変数の意味を日本語でメモする

といった**アウトプットをしながら読む**と、理解が定着しやすくなります。


### 8. わからない部分は「何がわからないか」をはっきりさせる

- 「この添字 $ i $ は何を表しているのか？」
- 「この行列積は、どの次元を縮約しているのか？」
- 「この softmax はどの軸に対して取っているのか？」

といった**具体的な疑問**を自分に問いかけながら読むと、どこで詰まっているかが明確になります。  
そのうえで、その疑問に答えてくれる解説記事や書籍を探すと効率的です。

## 総括

ニューラルネットワーク／LLM論文の数理モデルを理解するコツは、**「数式を設計図として読む」** ことです。

- 数式は、モデルの構造や計算手順を**正確に伝えるための設計図**です。
- 著者は、再現性・理論解析・実装のため、数式でモデルを定義します。
- 理解するときは：
  1. まず「この式は何をやっているのか？」を日本語でざっくりつかむ
  2. 次元（shape）を常に意識して、行列の行・列が何を表すか確認する
  3. 小さな具体例で手計算し、図やコードと対応づける
- こうすると、数式が「部品と配線の設計図」として見えてきて、読みやすくなります。

日本語と数理モデルを見比べっこして、まず初めになんとなくつかんでみて頂ければ。
その後、他の論文を読むと、同じような表記が使われています。
これを繰り返していくうちに、数理モデルの耐性が出来て来るのではと思っています。
