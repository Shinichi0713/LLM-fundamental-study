    良く使われる数理モデル

    LLM（大規模言語モデル）の数理モデルは、基本的に Transformer アーキテクチャ をベースにしているので、その論文で使われる表記がほぼ共通の「標準」になっています。

    代表的なものを整理すると、だいたい次のような表記が主流です。

    1. 入力・埋め込みまわりの表記

    • トークン列の長さ：
      • n n n や L L L（sequence length）
    • 埋め込み次元：
      • dmodel d_{\text{model}} dmodel​ あるいは単に d d d（hidden dimension）
    • 入力行列：
      • X∈Rn×d X \in \mathbb{R}^{n \times d} X∈Rn×d や X∈Rn×d \mathbf{X} \in \mathbb{R}^{n \times d} X∈Rn×d（トークン数×次元の行列）

    2. Attention（Q, K, V）まわりの表記

    • Query, Key, Value：
      • Q,K,V Q, K, V Q,K,V（それぞれ Query, Key, Value の頭文字）
      • あるいは Q,K,V \mathbf{Q}, \mathbf{K}, \mathbf{V} Q,K,V（太字で行列を表す）
    • Attention 重み（スコア）：
      • Attention(Q,K,V)=softmax(QK⊤dk)V \text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^\top}{\sqrt{d_k}}\right)V Attention(Q,K,V)=softmax(dk​​QK⊤​)V
      • ここで dk d_k dk​ は Key の次元（head dimension）
    • マルチヘッド Attention：
      • MultiHead(Q,K,V)=Concat(head1,…,headh)WO \text{MultiHead}(Q, K, V) = \text{Concat}(\text{head}_1, \dots, \text{head}_h)W^O MultiHead(Q,K,V)=Concat(head1​,…,headh​)WO
      • headi=Attention(QWiQ,KWiK,VWiV) \text{head}_i = \text{Attention}(QW_i^Q, KW_i^K, VW_i^V) headi​=Attention(QWiQ​,KWiK​,VWiV​)
      • h h h はヘッド数、WiQ,WiK,WiV,WO W_i^Q, W_i^K, W_i^V, W^O WiQ​,WiK​,WiV​,WO は学習される重み行列

    3. 正規化・残差接続まわりの表記

    • Layer Normalization：
      • LayerNorm(x) \text{LayerNorm}(x) LayerNorm(x) あるいは LN(x) \text{LN}(x) LN(x)
      • 最近の LLM では RMSNorm もよく使われ、RMSNorm(x) \text{RMSNorm}(x) RMSNorm(x) と書かれることが多い
    • 残差接続：
      • x+Sublayer(x) x + \text{Sublayer}(x) x+Sublayer(x) あるいは x+F(x) x + F(x) x+F(x)
      • Pre-Norm か Post-Norm かで順序が変わる

    4. フィードフォワードネットワーク（FFN / MLP）

    • FFN の表記：
      • FFN(x)=W2 σ(W1x+b1)+b2 \text{FFN}(x) = W_2\,\sigma(W_1 x + b_1) + b_2 FFN(x)=W2​σ(W1​x+b1​)+b2​
      • あるいは MLP(x) \text{MLP}(x) MLP(x) と書く
      • 最近の LLM では SwiGLU など Gated Linear Unit 系が多く、
        FFN(x)=(W2 swish(W1x))⊙(W3x) \text{FFN}(x) = (W_2 \,\text{swish}(W_1 x)) \odot (W_3 x) FFN(x)=(W2​swish(W1​x))⊙(W3​x) のような形で書かれることも多い

    5. 位置エンコーディング（Positional Encoding）

    • 絶対位置エンコーディング（sin/cos）：
      • PE(pos,2i)=sin⁡(pos/100002i/d) PE_{(pos, 2i)} = \sin(pos / 10000^{2i/d}) PE(pos,2i)​=sin(pos/100002i/d)
      • PE(pos,2i+1)=cos⁡(pos/100002i/d) PE_{(pos, 2i+1)} = \cos(pos / 10000^{2i/d}) PE(pos,2i+1)​=cos(pos/100002i/d)
    • RoPE（Rotary Positional Embedding）：
      • 回転行列 RΘ,m R_{\Theta, m} RΘ,m​ を使って qm=RΘ,mWqxm q_m = R_{\Theta, m} W_q x_m qm​=RΘ,m​Wq​xm​ のように書かれることが多い

    6. 出力・損失まわりの表記

    • 出力確率分布：
      • P(wt∣w<t)=softmax(Woht) P(w_t \mid w_{<t}) = \text{softmax}(W_o h_t) P(wt​∣w<t​)=softmax(Wo​ht​)
      • ここで ht h_t ht​ は時刻 t t t の隠れ状態、Wo W_o Wo​ は出力重み
    • 損失（クロスエントロピー）：
      • L=−∑tlog⁡P(wt∣w<t) \mathcal{L} = -\sum_t \log P(w_t \mid w_{<t}) L=−∑t​logP(wt​∣w<t​)

    • 層のインデックス：
      • ℓ=1,…,L \ell = 1, \dots, L ℓ=1,…,L（層数）
    • ヘッドのインデックス：
      • i=1,…,h i = 1, \dots, h i=1,…,h
    • トークン位置：
      • t t t や i,j i, j i,j（i i i 番目のトークン、j j j 番目のトークン）

    • Q, K, V（Query, Key, Value）

    • softmax（Attention の重み計算や出力確率）
    • FFN / MLP（フィードフォワードネットワーク）

    • クロスエントロピー損失 L \mathcal{L} L

    著者の意図

    著者も趣味や好きだから数理モデルを使っているわけではないと考えてます。 LLM論文の著者が数理モデルを使う理由は、大きく分けて次のようなものです。

    1. モデルの構造を正確に定義するため

    LLMは複雑なニューラルネットワークなので、言葉だけでは構造を厳密に伝えられません。

    • 「Attention で各トークンが他のトークンを見る」と口で言っても、
      どういう計算で「見る」のかは曖昧です。
    • 数式で

      Attention(Q,K,V)=softmax(QK⊤dk)V\text{Attention}(Q,K,V) = \text{softmax}\left(\frac{QK^\top}{\sqrt{d_k}}\right)V Attention(Q,K,V)=softmax(dk​​QK⊤​)V

      と書けば、どの行列をどう掛けて、どこで softmax を取るかが一意に決まります。

    つまり、「何をどう計算しているか」を正確に共有するために数理モデルが必要です。

    2. 再現性（Reproducibility）を確保するため

    研究コミュニティでは、他の研究者が同じ結果を再現できることが重要です。

    • 数式でモデルを書いておけば、
      誰でも同じ式を実装して同じ構造のモデルを構築できます。
    • もし言葉だけで「こういう感じのAttentionです」と書くと、
      解釈の幅が広すぎて、再現実験が難しくなります。

    数理モデルは、「この論文のモデルはこういう式で定義されている」という共通仕様の役割を果たします。

    3. 理論的な解析や証明の土台にするため

    LLMの振る舞いを理論的に理解したいとき、数式がないと議論が進みません。

    • 例えば「層を深くすると勾配がどうなるか」「Attention の表現力はどれくらいか」といった理論解析は、
      数理モデルを前提に行われます。
    • また、新しい正規化手法や Attention の変種を提案するとき、
      「なぜこれがうまくいくのか」を数学的に説明する必要があります。

    数理モデルは、理論的な主張を支える共通言語として機能します。

    4. アルゴリズムや実装を明確に伝えるため

    LLMの学習・推論アルゴリズムも、数式で書くと明確になります。

    • 例えば「KVキャッシュを使った高速推論」を説明するとき、
      どのベクトルをキャッシュし、どのタイミングで更新するかを数式で書くと、
      実装者が迷わずコードに落とし込めます。
    • また、バッチ処理や並列化の都合で、
      どの次元をどう並べ替えるかといった実装上の工夫も、
      数式で表現すると理解しやすくなります。

    5. モデルのバリエーションや一般化を議論するため

    「標準的なTransformer」から少し変えた新しいアーキテクチャを提案するとき、
    どこをどう変えたかを数式で示す必要があります。

    • 例えば「QK-Norm を導入した」「RMSNorm を使った」「RoPE をこう変形した」などは、
      数式で差分を書かないと、読者に伝わりません。
    • また、一般化された形（例：任意のヘッド数・次元数）で書いておけば、
      別の設定（モデルサイズやタスク）に応用しやすくなります。

    6. コミュニティ内の共通言語として使うため

    LLM研究コミュニティでは、

    • Q,K,VQ, K, VQ,K,V
    • LayerNorm / RMSNorm
    • FFN / MLP
    • RoPE
    • Cross-entropy loss

    といった記号や略語が共通語になっています。

    数理モデルを使うことで、

    • 論文同士の比較がしやすい
    • 既存研究の式を引用しやすい
    • 新しい提案が既存手法とどう違うかが明確になる

    というメリットがあります。

    7. 実装者（エンジニア）に設計意図を伝えるため

    LLM論文は、研究者だけでなく、実際にモデルを実装するエンジニアも読者です。

    • 数式を見れば、どの部分が計算のボトルネックになりそうか
    • どの演算がメモリを多く使うか
    • どの部分を最適化すべきか

    といった設計上の勘所が読み取れます。

    言葉だけの説明だと、こうした実装上のポイントがぼやけてしまいます。

    理解する上でのコツ

    LLMの数理モデル（Transformer系）を読んで理解するコツは、だいたい次のような順番で進めると良いのではと思っています。

    1. まず「何をやっているか」を日本語でざっくり理解する

    数式をいきなり追いかけるのではなく、その式が何を実現しようとしているかを先に押さえます。

    • Attention なら「各トークンが他のトークンをどれだけ参照するかを決めて、その重み付き和を取る仕組み」
    • LayerNorm なら「各トークンのベクトルを平均0・分散1付近に揃えて、学習を安定させる」
    • FFN なら「各トークンごとに非線形変換をかけて表現を豊かにする」

    というように、「この式は何をやっているのか？」 を一言で言えるようにしておくと、後で数式を追いやすくなります。

    2. 次元（shape）を常に意識する

    LLMの数式は、行列やベクトルの演算がほとんどです。
    どの次元が何を表しているかを意識すると、理解が一気に楽になります。

    • X∈Rn×d X \in \mathbb{R}^{n \times d} X∈Rn×d：
      • n n n：トークン数（文の長さ）
      • d d d：隠れ次元（埋め込み次元）
    • Q=XWQ∈Rn×dk Q = X W_Q \in \mathbb{R}^{n \times d_k} Q=XWQ​∈Rn×dk​：
      • dk d_k dk​：1ヘッドあたりの次元（head dimension）
    • Attention のスコア行列：
      • QK⊤∈Rn×n QK^\top \in \mathbb{R}^{n \times n} QK⊤∈Rn×n（各行が「あるトークンが他の全トークンにどれだけ注目するか」）

    といった具合に、「この行列の行は何？列は何？」 を常に頭の中で確認しながら読むと、式の意味がはっきりします。

    3. 小さい例（具体例）で手計算してみる

    論文の式は一般形で書かれているので、自分で小さな例を作って手計算すると理解が深まります。

    • トークン数 n=3 n = 3 n=3、次元 d=4 d = 4 d=4 くらいの小さな行列 X X X を適当に作る
    • WQ,WK,WV W_Q, W_K, W_V WQ​,WK​,WV​ も小さな行列を仮定して、Q,K,V Q, K, V Q,K,V を計算してみる
    • QK⊤ QK^\top QK⊤ を計算して、softmax をかけてみる
    • その重みで V V V を加重平均してみる

    実際に数値を入れてみると、「あ、これが Attention の重み行列なんだな」「この列が各トークンの出力になるんだな」と腑に落ちます。

    4. 図（アーキテクチャ図）と数式を対応づける

    多くの論文には Transformer のブロック図が載っています。

    • 図の中で「ここが Q」「ここが K」「ここが V」と対応づける
    • 図の「Add & Norm」が、数式では x+LayerNorm(F(x)) x + \text{LayerNorm}(F(x)) x+LayerNorm(F(x)) に対応する、といった具合に

    図を眺めながら数式を読むと、どの式がどのブロックに対応しているかが分かりやすくなります。

    5. 1ブロックだけに集中して理解する

    LLM は同じブロックを何十層も積み重ねたものです。
    一度に全部を理解しようとせず、

    • 1つの Transformer ブロック（Self-Attention + FFN + LayerNorm + Residual）だけに絞る
    • そのブロックの入出力の関係を完全に理解する
    • それが分かれば、あとはそれを何層も積んだだけ、と考える

    というように、局所的な構造を徹底的に理解するのがコツです。

    6. 実装（PyTorch など）と見比べる

    最近の論文は1/2くらいの確率で論文を試行したコードをgithubで公開しています。 論文の数式だけでは分かりにくい部分も、公開されたコードを見ると一気に分かることが多いです。

    • Hugging Face の Transformers ライブラリや、PyTorch の Transformer 実装
    • あるいは「Transformer from scratch」系の解説記事のコード

    を見ながら、

    • 論文の式のどの部分が、コードのどの行に対応しているか
    • 次元の変換（view, permute, matmul）がどう行われているか

    を確認すると、抽象的な数式が具体的な操作として理解できます。

    7. 自分で図やメモを書く

    読むだけだと頭に入りにくいので、

    • 自分なりのブロック図を描く
    • 式の変形をノートに書き写す
    • 各変数の意味を日本語でメモする

    といったアウトプットをしながら読むと、理解が定着しやすくなります。

    8. わからない部分は「何がわからないか」をはっきりさせる

    • 「この添字 i i i は何を表しているのか？」
    • 「この行列積は、どの次元を縮約しているのか？」
    • 「この softmax はどの軸に対して取っているのか？」

    といった具体的な疑問を自分に問いかけながら読むと、どこで詰まっているかが明確になります。
    そのうえで、その疑問に答えてくれる解説記事や書籍を探すと効率的です。

    総括

    ニューラルネットワーク／LLM論文の数理モデルを理解するコツは、「数式を設計図として読む」 ことです。

    • 数式は、モデルの構造や計算手順を正確に伝えるための設計図です。
    • 著者は、再現性・理論解析・実装のため、数式でモデルを定義します。
    • 理解するときは：
      1. まず「この式は何をやっているのか？」を日本語でざっくりつかむ
      2. 次元（shape）を常に意識して、行列の行・列が何を表すか確認する
      3. 小さな具体例で手計算し、図やコードと対応づける
    • こうすると、数式が「部品と配線の設計図」として見えてきて、読みやすくなります。
